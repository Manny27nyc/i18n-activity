How to contribute to an Language Enablement group

This page gives advice about how to participate in a group working on language enablement. We'll refer to these as lreq groups or task forces.

This document serves as an entry point for those wanting to understand how they would participate in an existing group. It is intentionally brief, and its a good idea to first read Setting up a Gap Analysis Project. Although that page describes setting up a new group, it introduces some of the more general aims.

Each lreq group does its work in a GitHub repository. The repository has a home page, an issue list, and usually one or more documents to capture gaps and explain requirements.

Some groups produce documents for a single language or script, while other groups will generate multiple documents, each of which addresses one of a number of languages and/or scripts.

Goals

The goals of an lreq group include the following:

  • Investigate and document situations where people who are trying to use the Web in their own language and writing system encounter obstacles. This work should consider whether current standards and implementations provide, or promise support for, the problematic features, and should demonstrate the issues using tests or examples.
  • Provide a set of requirements for missing features, or in some cases describe more fully how the language/script should work. This is done in a technology agnostic way.
  • Constitute an expert network that the W3C can turn to if they have questions about internationalization of their technologies, or that can raise issues and requirements related to Web technologies.
  • Raise bugs and change requests against specifications and applications to encourage support for the features that end users are missing.

Roles

An lreq group needs participants to take on a number of roles. For more information and to help you decide which role you would like to play, see Layout task force roles.

GitHub discussions

All technical discussion should take place in GitHub issues.

It's ok to use languages other than English in a GitHub thread, but important information should eventually be summarised in English for the benefit of those who don't speak the language.

Those new to GitHub may wish to read a short introduction: Working with GitHub issues.

Describing gaps

The key deliverable of an lreq group is one or more gap-analysis documents. See the Japanese gap-analysis document, for example.

The W3C staff contact or the chair of the group will provide an initial template for the gap-analysis document. It uses a standardised set of headings down to the 3rd level, and then the detailed content is added topic by topic. There is a set of ideas at the beginning of each section, in italics, to suggest things to cover in a given section.

The detailed content of the gap-analysis document is produced using GitHub issues, rather than by editing the HTML. See an example topic for Japanese gap-analysis. The text of the initial comment in an issue is automatically inserted into the HTML document framework. The comment fields lower down the GitHub thread can be used to discuss the current text and/or provide alternative content.

For more details about how to produce a gap-analysis document, see How to contribute to a gap-analysis document.

The gap-analysis document aims to describe issues and the current state of implementation in specifications or browsers. It's not meant to describe the actual requirements – that is done in a separate document that is technology agnostic. These requirements documents can range from the very large Japanese Layout Requirements, to small documents that only provide the essential information needed to support the issues raised in the gap-analysis document, such as the Khmer Layout Requirements. These documents are written in HTML.

Tests

An important part of gap-analysis work involves exploring what features are supported and creating tests that can be used to communicate current behaviour to others.

The language enablement framework we use for our lreq groups provides a way to quickly create tests and tweak them to explore browser support. For more information about how to create these tests, see Writing i18n tests, and follow the link to the description of Interactive exploratory tests.
